# 需求管理重构需求文档

## 1. 背景

当前需求管理只支持项目下维护需求基础信息，并承接 XMind 用例上传流程。新的设计希望把需求从“静态列表”升级为“交付看板”，围绕需求采集、评审、分配、开发、测试、验收和完成形成可追踪的交付闭环。

本阶段在保留现有项目、需求、用例集、Bug、用户、角色权限能力的基础上，补充需求主流程状态、任务池、状态门禁和看板化展示能力。

## 2. 目标

- 支持需求看板按状态分组和筛选。
- 支持新建需求后自动进入“需求采集”。
- 支持需求维护提出人、提出日期、提出迭代、计划上线迭代和优先级。
- 支持需求主流程按固定阶段流转。
- 支持开发阶段按角色或人员拆分并行任务。
- 支持测试阶段按模块或测试人拆分并行任务。
- 支持所有开发任务完成后才允许进入测试。
- 支持所有测试任务完成后才允许进入 UI 验收。
- 支持 UI 验收、产品验收后进入完成。
- 支持需求详情展示基础信息、主流程轨迹、任务池和流转门禁。
- 保留需求与现有用例集、Bug 的关联入口。

## 3. 用户角色

### 管理员

- 查看全部需求。
- 创建和编辑需求。
- 维护任务池。
- 强制调整需求状态。
- 管理角色、权限和数据范围。

### 产品人员

- 创建需求。
- 维护需求基础信息。
- 推进需求采集、预评审、详细评审、分配、产品验收。
- 查看需求交付进度。

### 测试人员

- 查看参与需求。
- 在需求测试阶段维护测试任务。
- 上传或维护用例集。
- 在需求下创建 Bug。
- 推进测试任务完成。

### 开发人员

- 查看参与需求。
- 维护自己的开发任务状态。
- 查看需求下 Bug。
- 推进开发任务完成。

## 4. 功能需求

### 4.1 需求看板

看板顶部展示：

- 页面标题：需求看板。
- 当前筛选结果数量。
- 搜索框。
- 新建需求按钮。
- 状态筛选栏。

需求卡片展示：

- 需求名称。
- 需求说明。
- 需求编号。
- 提出人。
- 提出迭代。
- 计划上线迭代。
- 当前状态。
- 门禁状态。
- 优先级。

状态筛选支持：

- 全部。
- 需求采集。
- 需求预评审。
- 需求详细评审。
- 需求分配。
- 需求开发。
- 需求测试。
- UI验收。
- 产品验收。
- 完成。

搜索范围：

- 需求编号。
- 需求名称。
- 需求说明。
- 提出人。

### 4.2 新建需求

新建需求字段：

- 需求名称，必填，最长 36 字。
- 提出人，必填。
- 提出日期，必填。
- 提出迭代，必填。
- 计划上线迭代，必填。
- 优先级，默认 `P2`。
- 需求说明，选填，最长 120 字。

创建规则：

- 后端自动生成或校验需求编号。
- 创建人取当前登录用户。
- 初始主状态为“需求采集”。
- 初始任务池为空。
- 写入需求状态历史。

### 4.3 需求详情

详情区展示：

- 需求名称。
- 当前状态。
- 需求说明。
- 需求编号。
- 提出人。
- 提出日期。
- 提出迭代。
- 计划上线迭代。
- 优先级。
- 主流程进度。

主流程进度按固定顺序展示：

1. 需求采集。
2. 需求预评审。
3. 需求详细评审。
4. 需求分配。
5. 需求开发。
6. 需求测试。
7. UI验收。
8. 产品验收。
9. 完成。

### 4.4 开发任务池

需求进入“需求开发”后，可维护开发任务。

开发任务字段：

- 任务名称。
- 任务角色：服务端、iOS、安卓、小程序、鸿蒙、Web、其他。
- 负责人。
- 任务状态。
- 备注。

开发任务状态：

- 待排期。
- 待开发。
- 开发中。
- 联调中。
- 开发完成。

门禁规则：

- 开发任务池为空时，不允许进入“需求测试”。
- 全部开发任务状态为“开发完成”后，才允许进入“需求测试”。

### 4.5 测试任务池

需求进入“需求测试”后，可维护测试任务。

测试任务字段：

- 任务名称。
- 任务角色：测试。
- 负责人。
- 任务状态。
- 备注。

测试任务状态：

- 待排期。
- 待测试。
- 测试中。
- 测试完成。

门禁规则：

- 测试任务池为空时，不允许进入“UI验收”。
- 全部测试任务状态为“测试完成”后，才允许进入“UI验收”。

### 4.6 状态流转

需求主状态支持正向推进和回退。

正向流转：

- 需求采集 -> 需求预评审。
- 需求预评审 -> 需求详细评审。
- 需求详细评审 -> 需求分配。
- 需求分配 -> 需求开发。
- 需求开发 -> 需求测试。
- 需求测试 -> UI验收。
- UI验收 -> 产品验收。
- 产品验收 -> 完成。

回退规则：

- 完成前允许回退到上一状态。
- 回退不自动删除任务池。
- 回退需要写入状态历史。
- 从测试回退到开发后，原测试任务保留但不参与开发门禁。

### 4.7 用例和 Bug 关联

需求仍作为用例集和 Bug 的上游实体。

- 需求详情保留上传 XMind 和查看用例集入口。
- 需求详情可展示需求下 Bug 数量。
- 需求完成不自动关闭 Bug。
- 如存在未关闭 Bug，产品验收进入完成时需要提示，但第一阶段不强制拦截。

### 4.8 权限需求

建议新增或复用权限点：

- `MENU_REQUIREMENT`：访问需求管理。
- `REQUIREMENT_VIEW`：查看需求。
- `REQUIREMENT_CREATE`：创建需求。
- `REQUIREMENT_EDIT`：编辑需求基础信息。
- `REQUIREMENT_TRANSITION`：流转需求主状态。
- `REQUIREMENT_TASK_EDIT`：维护需求任务。
- `DATA_ALL`：查看全部数据。

权限规则：

- 管理员拥有全部权限。
- 非 `DATA_ALL` 用户默认只能查看自己创建或参与的需求。
- 任务负责人可以维护自己的任务状态。
- 状态流转需要 `REQUIREMENT_TRANSITION` 权限。

## 5. 状态和枚举

### 需求主状态

- `COLLECTING`：需求采集。
- `PRE_REVIEW`：需求预评审。
- `DETAIL_REVIEW`：需求详细评审。
- `ASSIGNING`：需求分配。
- `DEVELOPING`：需求开发。
- `TESTING`：需求测试。
- `UI_ACCEPTING`：UI验收。
- `PRODUCT_ACCEPTING`：产品验收。
- `DONE`：完成。

### 需求优先级

- `P1`：高。
- `P2`：中。
- `P3`：低。

### 任务类型

- `DEV`：开发任务。
- `TEST`：测试任务。
- `ACCEPTANCE`：验收任务，预留。

### 开发角色

- `BACKEND`：服务端。
- `IOS`：iOS。
- `ANDROID`：安卓。
- `MINI_PROGRAM`：小程序。
- `HARMONY`：鸿蒙。
- `WEB`：Web。
- `OTHER`：其他。

## 6. 非目标

第一阶段不做：

- 复杂 BPMN 工作流引擎。
- 多级审批和会签。
- 自动同步外部项目管理系统。
- 自动生成开发任务。
- 任务工时统计。
- 甘特图和资源排期。
- 需求完成时自动关闭 Bug。
- 复杂通知策略。
