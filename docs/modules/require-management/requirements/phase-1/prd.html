需求管理重构需求文档

将现有需求管理从项目下的基础列表升级为需求交付看板，围绕需求采集、评审、分配、开发、测试、验收和完成形成可追踪的交付闭环。

模块：require-management 阶段：phase-1 前端：Vue 3 + Element Plus 后端：Spring Boot + MyBatis Plus

目录

1. 背景 2. 目标 3. 用户角色 4. 功能范围 5. 交付流程 6. 字段设计 7. 门禁规则 8. 权限设计 9. 接口范围 10. 验收标准 11. 非目标

1. 背景

当前系统中的需求管理主要承担项目下需求维护、XMind 用例上传和用例集入口能力。随着缺陷、用户、角色权限等模块接入，需求需要成为交付过程的主线实体，支撑研发、测试、验收多角色协作。

新设计要求页面呈现为需求看板，并通过主状态和任务池控制需求从采集到完成的推进过程。

2. 目标

  • 支持需求看板按状态分组、筛选和搜索。
  • 支持需求主流程状态流转。
  • 支持开发阶段按角色或人员拆分并行任务。
  • 支持测试阶段按模块或人员拆分并行任务。
  • 支持开发任务和测试任务的完成门禁。
  • 支持需求详情展示流程进度、任务池和流转结果。
  • 保留需求与用例集、XMind、Bug 的现有关系。

3. 用户角色

管理员

查看全部需求，维护任务池，调整状态，管理用户、角色、权限和数据范围。

产品人员

创建需求，维护基础信息，推进采集、评审、分配和产品验收。

测试人员

维护测试任务，上传用例，创建 Bug，推进测试完成。

开发人员

查看参与需求，维护自己的开发任务状态，处理需求下 Bug。

4. 功能范围

4.1 需求看板

  • 顶部展示页面标题、筛选结果数量、搜索框和新建需求按钮。
  • 状态筛选包含全部、需求采集、需求预评审、需求详细评审、需求分配、需求开发、需求测试、UI验收、产品验收、完成。
  • 需求卡片展示需求名称、说明、编号、提出人、提出迭代、上线迭代、状态、门禁状态和优先级。

4.2 新建需求

  • 新建后自动进入“需求采集”。
  • 必填字段包括需求名称、提出人、提出日期、提出迭代、计划上线迭代。
  • 优先级默认 P2。
  • 创建成功后写入需求历史。

4.3 需求详情

  • 展示需求基础信息和主流程进度。
  • 展示当前阶段对应任务池。
  • 展示状态流转操作和门禁文案。
  • 保留 XMind 上传、用例集查看和 Bug 关联入口。

6. 字段设计

6.1 需求字段

字段 说明 规则

需求名称 需求标题 必填，最长 36 字
提出人 需求提出人 必填
提出日期 需求提出时间 必填
提出迭代 需求进入池子的迭代 必填

6.2 任务字段

字段 说明 规则
任务类型 开发任务、测试任务、验收任务预留 DEV、TEST、ACCEPTANCE
角色类型 服务端、iOS、安卓、小程序、鸿蒙、Web、测试、其他 用于分组展示和统计
负责人 系统用户 可为空，但页面提示责任不完整
任务状态 开发或测试状态 参与门禁判断

7. 门禁规则

门禁用于控制主流程正向流转。任务池本身可以并行推进，主状态只在全部任务满足条件后进入下一阶段。

7.1 开发门禁

  • 需求主状态为“需求开发”时启用。
  • 开发任务池为空时，不允许进入“需求测试”。
  • 所有开发任务状态必须为“开发完成”。

7.2 测试门禁

  • 需求主状态为“需求测试”时启用。
  • 测试任务池为空时，不允许进入“UI验收”。
  • 所有测试任务状态必须为“测试完成”。

7.3 回退规则

  • 支持回退到上一主状态。
  • 回退不删除任务池，也不重置任务状态。
  • 每次回退写入状态历史。

10. 验收标准

  • 使用管理员账号登录后，可以进入需求管理看板。
  • 创建需求后，需求状态为“需求采集”。
  • 需求可以按固定顺序正向流转。
  • 需求开发阶段没有开发任务时，不能进入测试。
  • 存在未完成开发任务时，不能进入测试。
  • 全部开发任务完成后，可以进入测试。
  • 需求测试阶段没有测试任务时，不能进入 UI 验收。
  • 全部测试任务完成后，可以进入 UI 验收。
  • 需求可流转到产品验收和完成。
  • 状态变化和任务变化可以在历史记录中追踪。
  • 需求下 XMind 上传、用例集查看和 Bug 创建能力不被破坏。
验证命令：后端执行 mvn -f pom.xml test，前端执行 npm --prefix frontend run typecheck。

11. 非目标

  • 不引入复杂工作流引擎。
  • 不做多级审批和会签。
  • 不自动同步外部项目管理系统。
  • 不自动生成开发任务。
  • 不做工时统计、甘特图和资源排期。
  • 需求完成时不自动关闭 Bug。
